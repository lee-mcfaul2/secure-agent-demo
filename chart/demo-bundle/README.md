## What this is

`bundle.tgz` is a **committed, unsigned demo-only prompt bundle** (envelope
schemas + shared schemas + per-MCP tool schemas + bundle manifest) conforming
to the v1.0 lib-agent-prompt bundle contract that the gateway's
`BundleView.from_bundle`, `MCPRegistry.from_bundle`, `ScrubCatalog.from_bundle`
and `SchemaRegistry` consume. The `bundle-fetcher` pre-install Job does
`tar -xzf bundle.tgz -C /bundle` (no `--strip-components`; entries are
top-level `./schemas/вҖҰ`, `./bundle-manifest.json`) into the `prompt-bundle`
PVC that the gateway, sandbox, and SQL-MCP mount.

## How it was generated

Built with lib-agent-prompt's own Go bundle-builder (`tools/bundle-builder`,
the engine behind `make bundle-example`), run against a source tree assembled
from the canonical repos in this workspace:

- `schemas/{user-prompt,final-response,tool-result}.json` and
  `schemas/shared/{error-envelope,pii-types,trace-context,uuid}.json` вҖ”
  copied **byte-identical** from `lib-agent-prompt/schemas/`.
- `schemas/services/agent-sql-mcp/*.json` вҖ” the 12 tool schemas
  (`get_order`, `list_orders`, `lookup_customer`, `search_customer` Г—
  `{request,response,meta}`) copied **byte-identical** from
  `agent-sql-mcp/schemas/`. `bundle-manifest.json` (version, cost caps,
  per-tool JCS digests) is generated by the builder from these.
- `schemas/shared/scrub-types.json` вҖ” **synthesized for this fixture.**
  lib-agent-prompt does not ship a `scrub-types.json` (the design-doc
  amendment renaming `pii-types.json` вҶ’ `scrub-types.json` with
  SECRET/CODEWORD families was never applied to that repo). The gateway's
  `ScrubCatalog.from_bundle` hard-requires `schemas/shared/scrub-types.json`
  and fails closed without it, so a demo catalog (PII for the customer
  fields agent-sql-mcp returns, plus secret/codeword backstops) is included
  here. **This file is the one piece not traceable to a canonical repo and
  should be replaced by lib-agent-prompt's real scrub catalog once that repo
  applies the amendment.**

Built reproducibly (`SOURCE_DATE_EPOCH=0`) and re-tarred with deterministic
metadata (`--owner=0 --group=0 --numeric-owner --mtime=@0`, sorted entries,
`./`-prefixed) to match the prior fixture's internal layout.

To regenerate: run the lib-agent-prompt `bundle-builder build` + `pack`
against an equivalent source tree, or re-run the assembly steps above.

It exists for **one reason**: in the real platform this PVC is populated by
**cosign-verifying and pulling the signed `lib-agent-prompt` OCI bundle** вҖ”
supply-chain integrity of the prompt/tool contract is a core point of the
whole system. That signed bundle is **not published**, so to keep
`helm install` turnkey we ship an unsigned, unverified fixture instead.

## Why this is NOT how you do it

The entire reason a signed bundle exists is so every component provably runs
the *same, attested* prompt/tool schemas. A committed tarball with no
signature and no verification throws that guarantee away вҖ” anyone can swap the
contents and nothing would notice. **This is the anti-pattern, with the lights
on**, exactly like `chart/demo-ca` and `chart/demo-secrets`.

## Extra caveat specific to this one

This fixture has been built to satisfy the **full v1.0 bundle contract** every
gateway consumer reads (verified by extract-and-assert: every required path
present and valid JSON). It is still **unsigned and unverified** вҖ” anyone can
swap the contents and nothing would notice. The seed Job remains **best-effort
and always exits 0**: if a future contract change outpaces this fixture, the
affected component fails on its own (localized, diagnosable) instead of
wedging the whole install at a pre-install hook. One file
(`schemas/shared/scrub-types.json`) is synthesized rather than sourced from a
canonical repo (see above).

## The correct fix

Publish the signed `lib-agent-prompt` v1.0 OCI bundle (fix that repo's release
pipeline), then restore the real `bundle-fetcher` that does
`cosign verify вҖҰ && oras pull вҖҰ` against it, and delete this directory.
